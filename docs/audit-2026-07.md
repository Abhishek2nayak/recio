# Vyooom — Full Codebase, Product & Market Audit

**Date:** July 3, 2026 · **Branch:** `feat/loom-parity-advanced` (large uncommitted diff included in audit)
**Scope:** ~26k lines across `extension/`, `server/`, `web/`, `shared/` + 18 DB migrations.

---

## Executive summary

The codebase is **substantially stronger than a typical pre-launch MVP**. The core architecture is sound and in several places genuinely well-engineered: bytes-never-touch-the-server uploads, true Drive resumable-protocol recovery with backoff, AES-256-GCM token encryption, instant share links, an IndexedDB crash-safety net for finished captures, revoked-grant detection, entitlement gating, and usage metering. The prompt-era concerns ("should we remove Excalidraw", "build a lightweight annotation system") are largely already resolved — a custom vanilla-JS draw/blur/overlay system exists and Excalidraw is confined to a lazy-loaded whiteboard page.

The real risks are elsewhere:

1. **First-run failure for free users** — the default upload destination is the *paid* Vyooom Cloud; a free user recording from the new on-page launcher hits an upgrade wall *after* recording (P0).
2. **In-memory recording buffer** — a browser crash or OOM mid-recording loses everything; there is no cap on duration and chunks are held in RAM (P0).
3. **WebM-only output** — share links are unreliable on iOS/Safari viewers, which breaks the product's core loop ("record → send to anyone") (P0 for a sharing product).
4. **Zero tests, no monitoring, no job queue** — the reliability scaffolding for a SaaS is absent (P1).
5. **Market reality check** — the "own your recordings" wedge is validated but no longer empty: Cap.so (open-source, 18k+ GitHub stars, BYO S3/self-host) owns the developer end of it. Vyooom's defensible slice is the *consumer-grade* version: "your Google Drive, zero setup, unlimited free recordings."

### 1.2 Recording lifecycle (current primary path)

1. Popup / shortcut → SW `SHOW_LAUNCHER` → content script mounts the **extension-origin launcher iframe** (`extension/src/launcher/main.ts`) — camera preview with MediaPipe background effects, live mic meter, device pickers, surface choice.
2. "Start recording" → SW creates the **offscreen document first** (listener-before-streamId race is explicitly handled, `background.ts:88-108`), shows the native `desktopCapture` picker, relays the stream id.
3. Offscreen engine (`offscreen/recorder.ts`) opens the stream, mixes mic + system audio through an `AudioContext`, records with quality-preset bitrates, ticks `RECORDING_TICK` every second (which also keeps the SW alive).
4. On-page dock mirrors state on every tab; live blur/draw tools mark up the page itself so the capture bakes them in; the camera bubble mounts only on the visible tab.
5. Stop → `finalize()`: fix WebM duration header → **persist blob to IndexedDB before any upload** → create metadata row + share link immediately (*instant link*, `uploadStatus: UPLOADING`) → poster thumbnail → direct-to-cloud bytes → `finalizeRecording` attaches the file id and applies the pending ACL → delete the local safety copy.

This is a faithful Loom-style pipeline and the instant-link design is the correct "magic moment" architecture.

## 2. Bugs & edge cases

Ordered by severity. File references are to current working-tree code.

### P0 — will hurt real users at launch

**2.1 Free-user first recording dead-ends at an upgrade wall.**
`auth.ts` gives every new user a FLOWCAP connection with `isDefault: true`; the extension's `Settings.destination` default is `"FLOWCAP"` (`extension/src/lib/storage.ts:87`); FREE plan has `hostedStorage: false`. The new on-page launcher has **no destination UI and no pre-flight check** — a free user who signs up, opens the launcher, and records gets `UPGRADE_REQUIRED` at *stop* time (`media-service.ts:224`). The recording survives in the pending store, but the first-run magic moment becomes a paywall error. (The popup shows a "Vyooom Cloud needs Premium" warning — but the launcher is now the primary surface.)
*Fix:* pre-flight the destination before `START_RECORDING` (SW asks the API once, cached): if destination is gated and Drive/Dropbox isn't connected, show a blocking "choose where recordings go" step in the launcher. Also surface destination in the launcher permanently (this is the brand promise — say where bytes go, per the disclosed-destination principle).

**2.2 A crash during recording loses the entire recording.**
Chunks live only in the offscreen document's memory (`chunks: Blob[]`) until stop. Browser crash, GPU crash, extension update, or OOM mid-recording = total loss. The IndexedDB safety net only starts at `finalize()`.
*Fix:* append chunks to IndexedDB as they arrive (every `ondataavailable`, already firing at 1s cadence), keep a "recording in progress" marker, and offer recovery on next popup open. This is also the fix for §2.3.

**2.3 No duration/memory ceiling.**
Nothing caps recording length. At the "high" preset (8 Mbps) an hour is ~3.6 GB held in RAM — then `fixWebmDuration` and `savePending` momentarily hold second copies. Expect OOM crashes well before the 5 GB `MAX_FILE_BYTES` sanity guard.
*Fix:* stream chunks to IndexedDB (§2.2) + a soft cap (e.g. 2h) with an on-dock warning.

**2.4 WebM-only output breaks viewers on iOS/Safari.**
MediaRecorder emits VP8/VP9 WebM; share pages serve the original bytes through the proxy to a native `<video>`. iOS Safari's WebM support is historically absent/partial — the person you send the link to on an iPhone often can't watch. For a product whose loop is "record → share with anyone," this is P0.
*Fix options (in order of pragmatism):* (a) prefer `video/mp4;codecs=h264` in `pickMimeType()` when `MediaRecorder.isTypeSupported` says yes (Chrome ≥ 126 supports mp4 muxing); (b) server-side ffmpeg transcode job for webm → mp4 as a fallback (needs the job queue from §5); (c) at minimum, detect unsupported playback on the share page and show a download/"open in browser" path.

### P1 — correctness bugs found in code

**2.5 `restart()` can corrupt the new take** (`offscreen/recorder.ts:184-202`). The old recorder's final `ondataavailable` fires asynchronously *after* `chunks = []` reassignment; its closure writes to the module-level `chunks` binding, so a stale mid-stream cluster from the discarded take becomes the *first* bytes of the new take (before the new WebM header). Players may reject or glitch.
*Fix:* also null `ondataavailable` before `stop()`, or capture chunks per-take in a local array keyed to the active recorder.

**2.6 SW in-memory state doesn't survive service-worker restarts** (`background.ts:23-31`). `uploads`, `recording.{active,studioTabId,viaOffscreen}` are module globals. RECORDING_TICKs keep the SW alive in the happy path, but if the offscreen doc crashes (no more ticks) the SW dies ~30s later and: the dock's `RECORDING_CONTROL` routes to `studioTabId: null` (controls dead), `GET_UPLOADS` returns empty (popup shows nothing mid-upload).
*Fix:* mirror durable state to `chrome.storage.session`; treat the in-memory map as a cache.

**2.7 Countdown is open-loop.** The offscreen engine `await delay(3 * 800)` while the content script independently animates 3-2-1 at 800ms/tick (`recorder.ts:125`, `content.ts:224`). Any message latency desyncs "recording actually started" from the visual "1". Recording can begin while "1" is still on screen — the first ~300ms of narration gets clipped.
*Fix:* have the content script send a "countdown finished" message; start the recorder on receipt (with a 4s timeout fallback).

**2.8 Concurrent first uploads can create duplicate "Recio" folders.** `ensureFolder` (list → create) has no lock; two parallel session inits race. Cosmetic but confusing in the user's Drive.

**2.9 Stuck-`UPLOADING` rows are immortal.** If a user discards a pending upload (or just never retries), the instant-link row + live share link stay "processing" forever. The share page stops polling after a stall, but the row/link persist.
*Fix:* janitor job — expire `UPLOADING` rows older than e.g. 24h to a "failed" state; show that state on the share page.

**2.10 View counting is trivially inflatable.** `GET /share/:token` increments `viewCount` on every resolve (reload = +1). `ViewEvent` has session dedup; `media.viewCount` doesn't use it.
*Fix:* derive display count from `ViewEvent` uniques, or increment only when a new sessionId registers.

### P1 — security hardening

**2.11 `/share/notify` is an unauthenticated webhook relay** (`share.ts:367`). SSRF is properly constrained to Slack/Discord hosts, but any anonymous IP can pump messages through your server to any known webhook URL (spam laundering, your IP reputation).
*Fix:* `requireAuth` it — the sender is always a logged-in sharer anyway — plus a per-user rate limit.

**2.12 `/share/:token/translate` spends your Anthropic budget unauthenticated.** Cached per (recording, lang), but any public link ×  ~all caption languages × every public recording is attacker-triggerable spend, and it's not metered against the owner's AI minutes.
*Fix:* meter translations to the owner's allowance and/or add a per-IP/token limiter.

**2.13 Share-passcode brute force has too much headroom.** `POST /:token/unlock` is only behind the general 120 req/min/IP limiter — ~172k guesses/day/IP.
*Fix:* dedicated limiter (e.g. 10/min per token+IP) + optional lockout.

**2.14 Refresh tokens can't be revoked.** Stateless 7d JWT: logout clears the cookie but a stolen token stays valid. Acceptable for beta; add a session/refresh-token table (jti + revocation) before charging money.

**2.15 Toast/HTML injection surface.** `showToast` interpolates `title`/`error` into `innerHTML` (`content.ts:184-190`). Strings are extension/server-originated today, but error text can embed upstream content (e.g. filenames in provider errors). Cheap fix: `textContent` assignment instead of template HTML.

### P2 — edge cases to schedule

- **Camera bubble not captured** when recording a non-Chrome window or a monitor whose visible tab isn't the one hosting the bubble; ditto blur/draw (they live in the page DOM). Either composite camera into the stream in the offscreen document (canvas compositor already exists) or document the limitation in-product.
- **chrome://, Web Store, PDF tabs**: launcher/dock can't inject (handled via studio fallback for start — good), but if the user *navigates* to one mid-recording the dock and bubble vanish; the native "stop sharing" pill becomes the only control. Consider a `chrome.notifications` fallback control.
- **Picker left open >30s** may let the SW idle out mid-`chooseDesktopMedia`; the callback (and stream id) die silently. Rare; a `chrome.storage.session` "pending start" flag would cover it.
- **`webPublish.ts` (web-side recording) has no IndexedDB safety net** — whiteboard/web recordings die with the tab. Port the pending-uploads store or unify pipelines (§4.2).
- **Multi-monitor**: no surface-preview of what's actually being captured (native picker handles selection; fine for MVP).
- **Mic/camera device unplugged mid-recording**: no `devicechange` handling; audio silently drops. Detect + toast.
- **Offline at stop-time**: `createRecording` fails → pending store catches it (good) — but the error message says "retry from the popup," and the popup's retry needs auth + network; consider auto-retry with backoff when connectivity returns (`navigator.onLine` listener in SW).
- **Accessibility**: hand-rolled dock/launcher/toast DOM has no ARIA roles, no focus management; countdown has no reduced-motion variant. Web app is better (Radix-style primitives) but unaudited.
- **Dark mode**: theming exists (`ThemeMode`), applied to extension surfaces and web; share-page dark mode should be verified against overlays/text contrast.

## 3. Missing features

### Critical (before launch)
1. **Destination pre-flight + onboarding** (§2.1): first-run flow = sign in → connect Drive (or pick paid Cloud) → guided test recording.
2. **Crash-safe recording** (§2.2/2.3) — Loom's reliability bar; without it one lost 20-minute recording = churned user + bad review.
3. **iOS-playable output** (§2.4).
4. **Error monitoring + structured logs** (Sentry or equivalent in all three runtimes; pino on the server). You cannot operate a recorder product blind.
5. **Background job runner** (BullMQ/pg-boss): transcription currently runs inside the HTTP request (`ai-service.ts:generateTranscript`) — long videos will hit proxy/request timeouts; also needed for transcode (§2.4) and janitor (§2.9).
6. **Transactional email** (workspace invites, password reset — reset flow appears absent entirely; verify). Resend/Postmark + a couple of templates.
7. **Legal & store readiness**: privacy policy, ToS, Chrome Web Store listing assets, and a **Google OAuth verification plan** — `drive.file`-style scopes, verified-app review with videos. Lead time is weeks; start now. (Check which scope is requested; restricted scopes trigger a security assessment.)
8. **Account deletion + data export** (GDPR baseline; you hold OAuth tokens).

### High priority (fast follows)
- **Download button** on share page + library (bytes are the user's — make that tangible).
- **Folders/collections** in the personal library (only workspaces exist today).
- **Per-token share revocation UX** already exists; add "copy at current timestamp" and embed code.
- **Editor: stitch/re-record a section** (trim + smart cleanup exist; mid-video re-record is Loom's editing anchor).
- **Notification when someone views/comments** (email or extension badge) — retention driver.
- **Drive quota pre-check** before recording (API exists — `getStorageQuota` — unused as a guard).
- **Slack share as a proper Slack app** (webhook-paste UX is power-user-only).
- **Playback speed / captions toggle persistence** on the share page (verify; cheap wins).

### Nice to have
- GIF/animated preview in link unfurls (og:image is static today).
- Chapters from transcript (auto-chapters — you already have word timestamps).
- Custom domains for share links (Business).
- Viewer insights per-viewer (who watched, drop-off point — ViewEvent already stores watchedPct).
- Screenshot annotation editor (arrows/text on captured screenshots).

### Future vision
- Desktop app (Electron/Tauri) for system-wide capture + system audio on macOS, off-Chrome reliability.
- More providers: OneDrive, S3-compatible (this is Cap's home turf — do it when a customer pulls).
- Public API/SDK + Zapier.
- Team analytics dashboards; SSO/SCIM for Business.
- Local-only mode (never create a server row — pure device + Drive).

## 4. Technical debt

1. **Zero tests.** Priorities: (a) server route/service integration tests against a test Postgres (share visibility rules, entitlement gates, finalize/ACL ordering are the highest-value targets); (b) publish-pipeline unit tests with a mocked API (retry/reuse-mediaId paths); (c) one Playwright smoke: record (fake media stream) → link resolves → plays.
2. **739-line file duplicated byte-for-byte twice**: `web/src/components/recio/primitives.tsx` ≡ `extension/src/components/recio/primitives.tsx` (and `icons.tsx`, 345×2). Move to `shared/` (or a `@flowcap/ui` package). Divergence has probably already started.
3. **Two publish pipelines**: `extension/src/storage/publish.ts` (safety net, instant link) vs `web/src/lib/webPublish.ts` (neither). Extract the flow into `shared/` with pluggable storage/persistence adapters.
4. **Two recorder engines**: `offscreen/recorder.ts` (primary) and `recorder/screenRecorder.ts` + `Studio.tsx` (1,337 lines, legacy fallback + whiteboard host). Decide the studio's fate explicitly: keep it *only* as whiteboard host + non-injectable-page fallback, and delete its duplicated recording UI once the launcher path has survived a few weeks of dogfood.
5. **Naming strata**: MyLoom (repo dir) → FlowCap (`@flowcap/*`, DB enum `FLOWCAP`, bucket) → Recio (Drive folder name, `recio/` components, IndexedDB name) → Vyooom (brand). Before external users: freeze a mapping doc; rename the user-visible bits (Drive folder "Recio" should be "Vyooom"); leave DB enum values alone (migration pain > benefit) but comment them.
6. **God components**: `MediaDetail.tsx` (1,008), `Popup.tsx` (663), `SharePage.tsx` (631), `Studio.tsx` (1,337). Split when next touched; don't refactor for its own sake.
7. **In-memory rate limiting** — fine single-instance; swap store when you go multi-instance (note it in the file).
8. **`UNLOCK_ALL_FEATURES`** — add a loud boot warning when enabled with `NODE_ENV=production` (one `if` + `console.error`; cheap insurance).
9. **No CI** visible — add typecheck + lint + (new) tests on PR.
10. Not over-engineered overall — the vanilla content-script decision, the "inert until keys exist" pattern for AI/Stripe/Dropbox, and the entitlement matrix are appropriately sized. Resist adding abstraction layers now.

## 5. Product review — would people switch from Loom?

**The hook is real.** Loom post-Atlassian: free tier cut to 25 videos / 5 min / 720p, Business $18–24/user/mo, price increases announced. "Unlimited free recordings because they live in *your* Google Drive" is a genuinely differentiated answer to the #1 Loom complaint (limits + price), and the implementation honestly delivers it (bytes never touch your server; ACLs, playback, deletion all work against the user's Drive).

**What would stop a switcher today:**
1. First-run upgrade wall (§2.1) — fatal to the exact moment that converts.
2. A lost recording (§2.2) — Loom's reliability is why people tolerate its pricing.
3. Recipient on an iPhone can't play the video (§2.4) — the *sharer* gets embarrassed once and leaves.
4. Trust: an unknown extension asking for Drive access needs the verified-app badge, a plain-language privacy page ("we store metadata + encrypted tokens, never your video"), and ideally the narrower `drive.file` scope story.
5. No notifications → no reason to come back to the dashboard.

**Friction inventory:** sign-in → connect Drive is a two-OAuth gauntlet (combine: "Sign in with Google *and connect Drive* in one consent"); the launcher hides where files go (undermines the one differentiator — put the destination chip on it); webhook-paste Slack sharing is dev-grade.

**What already feels ahead of expectations for this stage:** instant links, live blur/draw during recording, MediaPipe camera effects, transcript search across the library, non-destructive trim/cleanup/overlays, translated captions, password links + expiry, per-video CTAs, team library, engagement analytics.

## 7. MVP definition (smallest thing people pay for)

**Free (the growth engine):** record unlimited to *your* Drive/Dropbox → instant link → share page that plays everywhere → viewer comments/reactions → library. Free must be flawless; it *is* the marketing.

**Paid trigger (Pro, ~$8–10/mo):** AI transcript + title/summary + library transcript search + smart cleanup + translated captions (all built ✅), password/expiry links (built ✅), custom branding + CTA (built ✅), Vyooom Cloud storage for people who don't want to connect anything (built ✅).

**Not MVP:** workspaces/Business tier (built — keep dark until Pro converts), more storage providers, desktop app, public API.

The gap between today and this MVP is **not features — it's the six Critical items in §3.** You are feature-rich and reliability-poor; invert that.

## 8. Excalidraw & annotation — verdict

The premise of the original question is already resolved in the code:

- **Recording-time annotation** is a custom, dependency-free system: `content/draw.ts` (202 lines, pen/arrow-style drawing), `content/blur.ts` (214 lines, live blur regions), dock integration — captured live into the video. This *is* the "lightweight Loom-style annotation system" the spec asks for. ✅ Keep.
- **Playback-time annotation** is the non-destructive `overlays` JSON (text/box/blur with time ranges) rendered by the players. ✅ Keep.
- **Excalidraw** survives only as the standalone **whiteboard-recording page** (`web/src/pages/Whiteboard.tsx`), lazy-loaded per route (no bundle cost to the core app), embedded by the studio's whiteboard mode.

**Recommendation:** keep Excalidraw exactly where it is, but treat whiteboard mode as a non-blocking side feature: don't extend it, don't let it gate launch, and cut it from marketing if it can't be dogfooded to quality. Removing it entirely saves little (it's lazy-loaded) and loses a real use case (talking over a sketch). What's still worth adding to the *core* annotation set, post-launch: arrows/shapes in `draw.ts` (it's freehand-centric today), click highlights, and a spotlight/dim tool — all fit the existing vanilla canvas approach; no library needed.

## 9. Roadmap

### Phase 1 — Stabilization (highest value per hour)
1. Destination pre-flight + launcher destination chip (§2.1)
2. Incremental IndexedDB chunk persistence + recovery-on-restart + duration cap (§2.2/2.3)
3. mp4-first `pickMimeType` where supported; share-page unsupported-codec fallback (§2.4)
4. `restart()` stale-chunk fix (§2.5) · SW state → `chrome.storage.session` (§2.6) · countdown handshake (§2.7)
5. Security shortlist: auth on `/share/notify`, unlock limiter, translate metering, toast `textContent` (§2.11–2.15)
6. Test harness bootstrap: server integration tests for share/entitlement/finalize paths; publish-pipeline unit tests; CI.

### Phase 2 — Production readiness
